# 2026-05-06 — Strategy 1.1.5: OB-top + FVG-macro + RDRB4-htf

Параллельная ветка к 1.1.1/1.1.6. Новый htf-примитив — **4-свечной RDRB**
(вместо OB или FVG как в других 1.1.x).

## Контекст

После live-integration 6 стратегий (утро) пользователь предложил новую
стратегию: те же top OB и macro FVG, но htf-зона = **4-candle RDRB**
вместо OB/FVG. Концепция RDRB как ловушки ликвидности с 4 свечами
вместо 3 классических.

## RDRB-4 геометрия (canon)

Свечи c1, c2, c3, c4 (по времени). c4 = триггерная (= idx).

**SHORT (зеркально для LONG):**
- `c1.low > c2.low` — c2 выносит ликвидность снизу c1
- `c1.low < c2.close` — c2 закрывается выше c1.low (ложный пробой)
- `c2.low < c4.high` — верхний фитиль c4 заходит в нижний фитиль c2
- `c3.close < c2.low` — c3 закрытие ниже c2.low (поглощение)
- `c1.low > c4.high` — c4 не пробивает c1.low

**Зона (расширенная — после правок пользователя):**
- SHORT: `[max(c2.low, c4_body_high), c1.low]` где `c4_body_high = max(c4.open, c4.close)`
- LONG: `[c1.high, min(c2.high, c4_body_low)]` где `c4_body_low = min(c4.open, c4.close)`

Логика: пересечение **нижнего фитиля c2** и **верхнего фитиля c4** даёт
нижнюю границу зоны для SHORT. Верх растягивается до `c1.low`.

Sanity-check на 22-апр-26 SHORT 21:00 UTC: `c1.low=78828, c4_body_high=78762,
c2.low=78706 → bottom=78762, top=78828`. ✓ совпало с расчётом пользователя
вручную.

## Каскад 1.1.5

```
top OB (1d / 12h)
  └── macro FVG (4h / 6h) внутри top OB (zones_overlap)
        └── htf RDRB-4 (1h / 2h) с расширенной зоной zones_overlap с FVG
              └── entry: касание ближнего края зоны
```

**Lookahead-prevention** (урок [[strategy-1-1-6-look-ahead-macro-htf]]):
- macro-FVG `search_start = top_ob.cur_time + top_tf_hours`
- htf-RDRB `c1_time >= fvg_macro.c2_time + macro_tf_hours`

Все 4 (macro_tf, htf_tf) комбинации валидны (4h+1h, 4h+2h, 6h+1h, 6h+2h).

## Entry / SL / TP

- **SHORT entry** = `max(c2.low, c4_body_high)` (нижний край зоны = первое касание снизу)
- **LONG entry** = `min(c2.high, c4_body_low)` (верхний край зоны)
- **SL SHORT** = `max(c1.high, c2.high)` (выше всей структуры RDRB)
- **SL LONG** = `min(c1.low, c2.low)`
- **RR = 1.0** (фикс, как 1.1.6 baseline)

Без буфера на SL. Без SWEPT-фильтра.

## Эволюция спеки в течение сессии

1. **Зона v1**: `[c4.high, c1.low]` — узкая полоска ($5-30 ширины).
   Backtest: raw=288, closed=18 (NOT_FILLED=105/135 = 78%!), WR=77.8%, +10R.
   **Узкая зона = высокий WR но почти нет fill.**

2. **Зона v2 (расширенная)**: `[max(c2.low, c4_body_high), c1.low]`.
   Backtest: raw=318, closed=91 (NOT_FILLED=0), WR=58.2%, +15R, R/tr=+0.165.

Промежуточные неправильные интерпретации спеки (отлажено итерациями
с пользователем): сначала я понял `entry = c2.low`, потом думал про `c1.low`,
потом `c2.close + c4.close` — пока пользователь не дал конкретный пример
с числами (22-апр SHORT) для калибровки.
