# 2026-05-06 — Strategy 1.1.6 первый прогон + найденный lookahead

Сессия: с нуля до research-фиксации новой стратегии 1.1.6 за один заход.
Использован пошаговый протокол: чтение образцов → код → ревью → тесты →
backtest → ревью результатов → фикс bug'а → vault. По правилам пользователя
ждать «ок» после каждого шага.

## Что сделано

### Шаг 1 — изучение образцов

Прочитан 1.1.1, 1.1.2, 1.1.3, 1.1.1 backtest, 1.1.1 tests. Решение:

- **Импортируем** atomic-хелперы из 1.1.1 (canon): `OBZone`, `FVGZone`,
  `detect_ob_pair`, `detect_fvg`, `zones_overlap`.
- **Пишем заново** высокоуровневые collect/find — геометрия 1.1.6
  принципиально инвертирована (FVG-OB-FVG vs OB-FVG-OB-FVG),
  адаптация `collect_valid_macro_fvgs` была бы запутаннее, чем
  переписать.
- **Никаких cross-research импортов** между research/-папками
  (правило 2026-05-04 рефакторинга).

### Шаг 2 — `strategies/strategy_1_1_6.py`

3 новых helper'а:
- `collect_valid_top_fvgs` — все FVG на 1d/12h
- `find_first_macro_ob_for_top_fvg` — earliest macro-OB после top-FVG,
  с wick-инвалидацией top-FVG в окне `[c2_close, ob.cur_close)` на
  df_macro
- `find_first_fvg_htf_in_zone` — earliest htf-FVG, double-overlap
  (macro AND top), без SWEPT

Главная функция `detect_strategy_1_1_6_signals(df_1d, df_12h, df_4h,
df_6h, df_1h, df_2h, verbose=False)` — 1m в детектор не передаётся,
fill-scan живёт в backtest.

**Earliest-wins:**
- macro: по `cur_close = cur_time + macro_hours` (учитываем разницу 4h vs 6h)
- htf: по `c2_close = c2_time + htf_hours` (учитываем 1h vs 2h)

В 1.1.1 earliest-wins для 15m vs 20m по `c2_time` без учёта htf_hours
(разница 5 мин пренебрежимо). У 1.1.6 разница 100% — учитываем явно.

**SL/TP/RR=1.0:**
- entry = mid htf-FVG
- LONG: sl=ob_macro.bottom, tp=entry+risk×RR
- SHORT: sl=ob_macro.top, tp=entry-risk×RR
- `RR=1.0` как модульная константа (single source of truth)

### Шаг 2.5 — ревью пользователя, 5 правок

1. **Wick-инвалидация: `df_top` → `df_macro`** — на 1d-сетке между
   search_start и ob.cur_close могло быть 0 баров (4h окно), wick-проверка
   не срабатывала. Перевёл на df_macro как в 1.1.1
   `collect_valid_macro_fvgs`.
2. **Earliest macro по cur_close**, не cur_time. При равных cur_time
   4h победил бы по open_time, но реально 4h cur закрывается раньше 6h
   только потому что короче — учитываем явно.
3. **`RR = 1.0` модульная константа**, не локальное значение.
4. **Подтверждено:** `detect_fvg` не может вернуть и LONG и SHORT для
   одной тройки (условия взаимоисключающие).
5. **Подтверждено:** двойной `zones_overlap` нужен (overlap не транзитивен).
6. **Параметр `df_top`** удалён из `find_first_macro_ob_for_top_fvg`
   (мёртвый после переключения на df_macro).

### Шаг 3 — `tests/test_strategy_1_1_6.py`

6 тестов, все на синтетических свечах:
1. happy-path LONG
2. happy-path SHORT
3. wick-инвалидация (low<fvg.bottom в окне)
4. htf-FVG не пересекает top-FVG (double-overlap отсёк)
5. earliest-wins macro по cur_close (6h побеждает 4h при таком геометрии)
6. RR=1 точное равенство `|tp-entry| == |entry-sl|`

Все 6 PASSED с первого прогона, без подгонки фикстур (геометрия
просчитана на бумаге заранее).

### Шаг 4 — `research/1_1_6/backtest/backtest_strategy_1_1_6.py`

Адаптация из 1.1.1 backtest:
- `simulate_outcome` с **no_entry-логикой**: до касания entry если
  TP или SL hit → outcome=NO_ENTRY (структура отработала без нас)
- 5 outcomes: WIN/LOSS/NO_ENTRY/NOT_FILLED/OPEN
- `tf_minutes = 60 if htf_tf=="1h" else 120` — выводится из метаданных
  (урок [[strategy-1-1-1-look-ahead-15min-vs-tf_duration]])
- `dedupe_signals` двухэтапный bucketing (по entry на primary +
  по SL внутри primary, обе 0.5%)
- **`must_match` минимальный** = `["outcome", "fill_time"]`. Изначально
  включал `htf_tf, htf_fvg_zone` — упало на реальном кейсе 2026-01-29
  (два пути top-1d→1h-FVG и top-12h→2h-FVG к одной (entry, sl) с
  разным `htf_tf`). Это легитимный кейс confluence, должен агрегироваться
  через sorted-uniq join (`"1h,2h"`), не блокировать assert.

### Шаг 5.5 — пользователь нашёл lookahead на TV

Ручная сверка сигнала 2026-02-14 LONG показала:
- ob_macro 6h cur_time = 02-16 21:00 → cur_close = 02-17 03:00
- htf 1h c2_time = 02-17 00:00 ← за 3 часа ДО закрытия cur 6h-OB

То есть «реакция» формировалась **до того**, как macro-OB существовал
как сформированная структура. Это не баг детектора макро или фрактала —
это баг параметра в `find_first_fvg_htf_in_zone`:

```python
# БЫЛО:
htf_hours = 1 if htf_label == "1h" else 2
htf_start = ob_macro.cur_time + pd.Timedelta(hours=htf_hours)

# ДОЛЖНО:
htf_start = ob_macro.cur_time + pd.Timedelta(hours=macro_hours)  # 4 или 6
```

Длительность бара выводилась из ТФ поиска (1h/2h), а должна — из ТФ
candidate-структуры (4h/6h).

**Идентичный класс ошибки** что [[strategy-1-1-1-look-ahead-15min-vs-tf_duration]].

Применил fix. Тесты упали (4/6) — фикстуры скрывали тот же баг.
Сдвинул htf-FVG свечи в фикстурах (с 09-11 на 12-14 для 4h-macro,
с 07-09 на 12-14 для 6h-macro). После этого все 6 PASSED.

## Главные insights

1. **Lookahead типа «бар не закрылся, а мы уже ищем реакцию» —
   повторяющаяся грабля.** Уже видели в 1.1.1 (15min vs tf_duration).
   Теперь в 1.1.6 (htf_hours vs macro_hours). Класс одинаковый:
   длительность бара выводится из ТФ candidate-структуры, не из ТФ
   поиска и не из контекста скрипта. **Добавить в known-pitfalls
   как 10-й пункт с обновлённым правилом избегания.**

2. **Тесты могут скрывать тот же баг, что и продакшен-код.** Если
   фикстуры строились на ошибочном понимании логики, они «зелёные
   ложно». При fix'е продакшена ожидать падения тестов и
   пересматривать фикстуры (а не подгонять их).

3. **Визуальная сверка с TV — критический шаг для новых стратегий.**
   Автотесты не покрывают такой класс ошибок: они проверяют конкретные
   числа на синтетических фикстурах, которые тоже могут скрывать
   эту же грабли. Нужна минимум одна независимая сверка реального
   сигнала на чарте.

4. **NO_ENTRY 86% — диагностический сигнал.** Mid-FVG entry слишком
   оптимистична для этой геометрии. Аналогичная проблема у 1.1.1
   решалась через `entry_pct=0.80` (Stage 1 optimize). Если
   возвращаться к 1.1.6 — это первый кандидат на vary.

5. **Lookahead-fix не улучшает edge, делает его честнее.** Шёл от
   −10R до −5R — это не улучшение стратегии, а удаление мусорных
   «магических» сделок. Метрики стали корректные, но edge всё ещё
   отрицательный.
