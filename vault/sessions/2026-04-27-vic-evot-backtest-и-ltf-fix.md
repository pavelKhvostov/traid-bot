## Insights

### BTC × multi-RR анализ показал бимодальность

Прогнал BTC на RR ∈ {2.0, 2.2, 3.0}: **win/loss split идентичен (7/12)
во всех трёх вариантах**. То есть все 3 winner-сделки доходили до 3R,
а 6 EOD-сделок не дотягивали даже до 2R. Промежуточных winners нет.

Структура сигналов **бимодальна**: либо цена идёт сильно (≥3R), либо
вообще не идёт. Expectancy линейно растёт: 0.105R → 0.179R → 0.474R.
**На этой выборке RR=3.0 чистый upside** — те же 7 winners забирают
по 3R вместо 2R, SL и EOD не меняются.

Caveat: n=19 (теперь 21 после fix LTF), малая выборка. Один outlier
с MFE 4-5R может сломать идентичность. Прогон на 6-12 месяцах нужен
для уверенности.

### maxV LTF — двухшаговый fix

Юзер показал Pine-скрипт ASVK ViC. На 1D-чарте с `auto=true, mlt=100`:
`LTF = 1440/100 = 14.4 min → timeframe.from_seconds(864)`.

**Шаг 1 (неверный, `f88ee54`):** предположил Pine rounds DOWN → 14m.
Прогон бэктеста: total=61, WR=60.7%. Стат вменяемая, закоммитил.

**Шаг 2 (верный, `b418f86`):** юзер прислал TV-значение maxV(BTC, 2026-04-26)
≈ 78416. Прогнал мои расчёты на LTF ∈ {1, 5, 10, 14, 15, 30, 60}m:
- 14m: 78581.50 (Δ +165, мимо)
- 5/10/15/30/60m: все 78417.41 (Δ +1.4, точное совпадение)

Pine `timeframe.from_seconds(864)` возвращает **closest valid TF**, не "rounds down".
Из {600s=10m, 900s=15m} ближе 900 (Δ36 vs Δ264) → Pine = 15m. Сменил
`VIC_LTF_MINUTES = 14 → 15`. Бонус: 15m уже native в проекте.

После fix: total=67 (+6), WR=62.7% (+2pp), SOL улучшился 44%→50%.

См. [[vic-maxv-расходился-с-pine-индикатором-из-за-1m-вместо-15m]] —
debugging-заметка с симптомом, причиной, решением и реперфикс-таблицей.

## Что узнал

- **«Closest» vs «rounds down» в Pine.** Документация v5 неоднозначна, но
  поведение `timeframe.from_seconds()` с 864s показало "closest valid"
  (15m), а не "next lower" (10m или 14m). Надо запоминать.
- **Сверять числами, не интерпретацией.** Сделал теоретический guess 14m
  по docs reading — оказался не прав. Один TV-снимок → за 30 секунд
  понятно что неверно. Скрин дешевле часа спекуляций.
- **Сигналы бимодальны на BTC.** Для строгих стратегий с фрактальным
  паттерном это, наверное, нормально: либо breakout срабатывает с большим
  движением, либо setup был ложный с самого начала. Промежуточных
  состояний меньше.
- **EOD-исходы стоят отдельной обработки.** Сейчас они считаются как
  win/loss бинарно по знаку P&L. Реальный P&L EOD-сделки в R-единицах
  где-то между −1 и +RR. Для честной expectancy нужно
  `(exit_close − entry) / risk` per row, а не RR. Не переделывал — пометил
  на следующий итерационный шаг.

## Открытые задачи на следующую сессию

Юзер озвучил две правки в backtest, не успели реализовать перед концом сессии:

1. **Убрать SOL** из 3-символьного списка. WR=50% (vs ETH 77.8%, BTC 66.7%) —
   подозрение что VIC хуже работает на high-volatility pump/dump паттернах
   SOL. Хочется чище baseline на BTC+ETH. Поменять `SYMBOLS` в
   [backtest_vic_evot.py:25](../../backtest_vic_evot.py#L25), либо вынести в CLI-arg.

2. **Дедуп сигналов в одном дне.** Сейчас если два разных 15m setup-а
   на одном дне срабатывают (i+2 в разное время) — оба попадают в CSV.
   Юзер хочет: после первого сигнала в дне D дальше не искать. То есть
   первый valid 15m в дне D → пишем в результат → break внутреннего цикла
   `for i in range(4, len(df_15m_day))` в `collect_signals`.
   Эквивалент live-логики — раз сигнал отправлен, дедуп через `was_sent`
   блокирует повторы того же `(strategy, symbol, source_tf, direction,
   confirm_iso)`. Но в backtest нет `was_sent` — нужен явный break per-day
   per-symbol.

3. **Live deploy** ([[2026-04-27-vic-evot-реализация]] упоминал) — на юзере.
   Ему нужно `del state\vic_levels.json && python main.py`.
