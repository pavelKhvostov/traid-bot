# VIC maxV расходился с Pine-индикатором ASVK ViC из-за 1m вместо 15m

## Симптом

Стратегия VIC_EVOT была реализована end-to-end (см. [[2026-04-27-vic-evot-реализация]]),
но юзер заметил: `maxV`, который считает мой Python, **не совпадает** с тем,
что рисует TradingView-индикатор «ViC ASVK» на его 1D-графике BTC.

Конкретное значение для 2026-04-26: TV-индикатор рисовал линию ≈ 78416,
мой код возвращал **78400.0**.

## Причина

Pine-скрипт ASVK ViC имеет настройку `Auto = true, mlt = 100`. Логика:

```pine
rs = auto ? tfC / mlt : tfS
res := timeframe.from_seconds(min(tfC, rs))
```

На 1D-чарте: `tfC = 86400, mlt = 100 → rs = 864 секунды`. Дальше Pine
вызывает `timeframe.from_seconds(864)` и получает **ближайший** валидный
TF из стандартного набора {1, 3, 5, **10**, **15**, 30, 45, 60, ...} минут.

Из {600s=10m, 900s=15m} ближе **900s = 15m** (Δ=36 vs Δ=264). То есть
Pine просит у Binance свечи на 15m через `request.security_lower_tf` и
ищет максимально-объёмную свечу среди bull (close>open) или bear (close<open),
maxV = close этой 15m-свечи.

В моём `vic_levels.calculate_vic_d` я ошибочно использовал **сырые 1m
свечи**, что эквивалентно `auto = false` (LTF = 1m). На 1m максимум
объёма приходится на одну минуту, на 15m — на агрегат из 15 минут.
Победитель — другая свеча, другое close.

## Двухшаговый fix

**Шаг 1, `f88ee54`:** Прочитал docs Pine v5 как «rounds down to next lower
valid» и предположил Pine = 14m (840s, ниже 864). Прогнал на 14m → maxV
для BTC 2026-04-26 = **78581.50** (Δ +165 от TV). Стат бэктеста: total=61,
WR=60.7% — формально вменяемо, закоммитил.

**Шаг 2, `b418f86`:** Юзер прислал точное TV-значение ≈78416. Прогнал
сравнительную таблицу maxV для BTC 2026-04-26 на разных LTF:

| LTF | maxV | Δ от TV |
|-----|------|---------|
| 1m | 78400.00 | −16.0 |
| 5m | 78417.41 | +1.4 |
| 10m | 78417.41 | +1.4 |
| **14m** | **78581.50** | **+165.5** ← мой f88ee54, мимо |
| **15m** | **78417.41** | **+1.4** ← match |
| 30m | 78417.41 | +1.4 |
| 60m | 78417.41 | +1.4 |

5m/10m/15m/30m/60m все сходятся на 78417.41 — это потому что у каждой
из этих гранулярностей peak-volume bar захватывает один и тот же
1m-фрагмент с close=78417.41. На 14m иначе: 1440/14 = 102.857 (не делит
сутки чисто), границы LTF-баров плывут относительно эпохи и захватывают
другую комбинацию 1m свечей.

Pine `timeframe.from_seconds()` возвращает **closest valid**, не
**rounds down**. На 864s → 15m (closer to 900 than 600).

Сменил `VIC_LTF_MINUTES = 14 → 15`. Сверка: мой 78417.41 ≈ TV 78416 ±1
(погрешность чтения линии с графика).

## Где живёт в коде

[config.py](../../../config.py) — `VIC_LTF_MINUTES = 15` с комментарием.

[vic_levels.py](../../../vic_levels.py) — `calculate_vic_d(df_1m, day,
ltf_minutes=1)` принимает `ltf_minutes` параметр. При `>1` ресемплит
1m → ltf_minutes через `pandas.resample(origin='epoch', label='left',
closed='left').agg({open:first, high:max, low:min, close:last,
volume:sum}).dropna(subset=['close'])` — выравнивание по UTC-эпохе как
у `data_manager.compose_from_base`.

[vic_scanner.py](../../../vic_scanner.py) и
[backtest_vic_evot.py](../../../backtest_vic_evot.py) вызывают с явным
`ltf_minutes=VIC_LTF_MINUTES`.

## Как проверять впредь

1. Перед коммитом fix-логики на основе Pine-скрипта — попросить юзера
   прислать снимок индикатора с конкретным значением. **Один скрин
   дешевле часа спекуляций по docs.**
2. Прогонять comparison-таблицу maxV по 5+ LTF-вариантам, не один.
3. Если 5m/10m/15m/30m/60m дают одинаковый ответ — это сильный сигнал
   что Pine использует один из них (а не нестандартный 14m). Это
   отбраковывает гипотезы из «rounds-down» лагеря.
