### Entry-bar lookahead: вход=close[j] + управление с того же бара = фантомные TP (WR 68%→36%)

Что было (2026-06-21, swing_test/mr_validate): вход = `close[j]`, а sim проверяла SL/TP с `range(ei, end)` —
включая бар j, чьи high/low УЖЕ сформированы к close-входу. → стратегия «срывала» TP на баре входа задним числом.
Симптом: MR-у-экстремумов +0.601R/PF2.9/WR68%/год7/7/null p0.000 — всё рухнуло в −0.179R/PF0.73/WR36%/год0/7/
null p0.987 после фикса `range(ei+1, end)`. Так же умер «победитель» Дв.3 sweep+reclaim (+0.216R→−0.05R).
Причина: если вход НЕ на open бара (а на close или на уровне, тронутом внутри бара), управление этим же баром
видит уже-случившееся движение. Правило: вход=close → управление СО СЛЕДУЮЩЕГО бара (ei+1); вход=open[i] →
можно с i (high/low после open); вход=limit-уровень внутри бара → как минимум пессимистичный тай-брейк, лучше ei+1.
ВАЖНО: permutation-null это НЕ ловит (null тоже на баговом сим; null ловит тайминг, не lookahead-сим). Эталон
чистоты сессии — futures_tradeability (вход=open[i1+1]). Любой WR≥65%/PF≥2.5 на наиве → искать ЭТОТ баг.
См. [[htf-sweep-reclaim-reversal-engine]].

### Идеальный/невозможный WR (100% или 0%) = немедленно искать инверсию знака уровня/стопа

Что было (2026-06-21, b8_range_fade_v2): бэктест выдал WR=100% PF=9.90 (все сделки в плюс), потом после
правки — WR=0% (все в минус). Оба раза — знаковый баг: `sl = lvl + d*BUFFER*a` для шорта (d=-1) ставил стоп
НИЖЕ уровня (в сторону профита) → мгновенный «профитный стоп»; `atrl = anchor + d*0.5*a` для шорта ставил
уровень фейда НИЖЕ anchor вместо ВЫШЕ. Симптом: экстремальный WR (≈100% или ≈0%) и exp≈±1R на КАЖДОМ бакете.
Правило: при d=±1 множителе направления для уровня/стопа/TP — проверять оба знака на бумаге (шорт: уровень
ВЫШЕ anchor, стоп ВЫШЕ уровня, TP НИЖЕ; лонг — зеркально). Любой WR>85% или <15% по всем бакетам = СТОП,
искать инверсию знака ДО выводов. Эталон корректной стороны — v1 без d-множителя (hi=anchor+R·a / lo=anchor−R·a).
См. [[vadim-integration-living-market-laws]].

### (Vadim) HTF-фичи читают FINAL close in-progress бара = lookahead до 72ч; ML AUC>0.65 = подозревать

Что было (ветка Vadim, ob_vc): `features/hma_at_entry.py` брал `closes[idx_at_event]`, где idx —
бар, СОДЕРЖАЩИЙ entry_ms (ещё формируется). Для 1d TF = чтение close в 24:00 при входе в 14:00 (10ч
будущего), для 3d — до 72ч leak. Симптом: production canon v3.3 (WR 72.4%, +1288R, AUC 0.79) оказался
на lookahead; после фикса реальный AUC 0.54, WR ~38%. Правило: HTF-фича при entry_ms младше close TF
ОБЯЗАНА использовать intraday partial-bar (closed_HTF_bars + 1m_close на entry_ms как virtual partial),
а не close содержащего бара. **Любой ML AUC > 0.65 на 14-day directional crypto — подозревать lookahead.**
Сходится с нашим (taxonomy/factor_rank: чистый OOS AUC у нас ~0.54-0.70 только под shuffle-контролем).
Источник: [[ob-vc-hma-features-lookahead-fix]] · [[ml-snapshot-not-trajectory]].

### Barrier-R edge ≠ торговоспособность нетто; фикс-косты убивают тонкий edge на низких ТФ

Что было (2026-06-21, futures_tradeability arc-паттернов): edge на симметричном ±1.5ATR барьере (rev_R,
fade_R и т.п.) — это ГРОСС и НЕ PnL. На фьючерсах гросс +0.052R/сделку превратился в нетто −0.046R после
taker 0.05%×2 + slip 0.02%×2 (0.14% RT) + funding. Симптом: «закон выживает все гейты», но нетто-минус.
Причина: косты — фиксированный %, а риск (стоп в %) масштабируется с ATR → на низких ТФ стоп мал, и 0.14% RT =
0.1-0.25R/сделку (съедает edge); на высоких ТФ стоп велик, косты пренебрежимы (1d нетто+, 1h ~0). Правило:
ВСЕГДА считать торговоспособность как нетто-PnL (entry/stop/TP + комиссии + funding + slip + пессимистичный
тай-брейк), сравнивать с random-entry null и прогонять cost-sensitivity 0/1/2×; barrier-R годен только для
СУЩЕСТВОВАНИЯ сигнала, не для денег. Тонкий гросс-edge → не standalone, а confluence/высокие ТФ/maker.
См. [[ta-curved-patterns-arcs-and-data-discovered-shapes]].

### Перекрытие окон раздувает n (autocorr) + кривизна/apex = моментум-конфаунд

Что было (2026-06-21, curves/shape_discovery/arc_analysis): скользящие окна с 50% перекрытием дают
десятки тысяч «наблюдений», но они автокоррелированы → bootstrap-p выглядит супер-значимым на ровном
месте. Симптом: p<0.001 при скромном edge. Правило: при перекрывающихся окнах НЕ доверять p как истине —
опираться на cross-asset (3/3 независимых рядов) + год-стабильность (независимые периоды); шаг окна ≥50%
длины; для строгого p — субсэмпл без перекрытия. Вторая грабля: в арках apex_pos ≈ наклон в правом крае
(deriv = 2a·L·(1−apex_pos)), поэтому «apex рано→продолжение / apex поздно→разворот» может быть просто
моментумом в точке завершения, а не свойством ДУГИ → нужен momentum-matched null (контроль на наклон),
иначе припишешь форме то, что даёт тривиальный моментум. См. [[ta-curved-patterns-arcs-and-data-discovered-shapes]].

### Стена красивых графиков = иллюзия отбора; «пробой по линии» имеет направленный bias

Что было (2026-06-19, нейро-модуль ТА-законов): юзер дал 7 аннотированных графиков «импульс→флаг→
продолжение» — выглядит законом. Объективно на 4671 архетипе: continuation НЕ работает (fwd_signed
−0.0055, хуже random, 69% reversed; against-флаг WR 8%). Симптом: уверенность из набора красивых
примеров, нарисованных В ТОЧКЕ пробоя со стрелкой-прогнозом (которая ещё не отработала). Причина:
selection/confirmation bias — приносят только сработавшие. Правило: «закон» = паттерн, чей edge
выживает после фальсификации (bias-free forward-return + permutation-null + cross-asset + оба режима +
симметрия UP/DOWN), а НЕ «N красивых примеров». Доп. грабля: меряя пробой коррекции по СТАТИЧЕСКОМУ
экстремуму пивотов, наклонный канал ложно «разворачивается» (нормальный ход внутри канала превышает
прошлый экстремум) → мерить по СПРОЕЦИРОВАННЫМ линиям + дублировать bias-free метрикой forward-return.
См. [[ta-chart-pattern-continuation-folklore]].

### Грид-бэктест: абсолютный ΣR завышается per-top-cap/свободным коллектором → ранжируй по per-trade+null

Что было (2026-06-19, cascade_grid): генерик-движок с PER_TOP_CAP=3 и свободным zone-join даёт счёт
сделок в 2-6× больше live-семейства → абсолютные ΣR (+246R и т.п.) НЕ сравнимы с live (1.1.2 +101R).
Симптом: грид-конфиг выглядит вдвое прибыльнее живой стратегии. Причина: больше сигналов на тот же
паттерн (cap) + более слабые окна/инвалидация, чем строгий live-скелет. Правило: при гриде/переборе
ранжируй и решай по **per-trade expectancy (ΣR/closed) + cross-asset перенос + год-стабильность +
permutation-null**, НЕ по абсолютному ΣR. Победителей грида обязательно прогонять через НАСТОЯЩИЙ
live-скелет (dedup, confirm_time, реальные окна) для честных объёмов перед любым live-решением.
См. [[cascade-grid-64-cross-asset-robust-shortlist]].

### Высокий avg-R при низкой частоте ≠ хорошая standalone-цепочка (для цепочки решает total R)

Что было (2026-06-19, вариант C RDRB-htf для 1.1.1): подмена htf-детектора OB→RDRB дала
заметно выше expectancy (BTC avg +0.466 vs +0.253 у OB-htf), но RDRB на 1h/2h в ~3.7× реже
(74 vs 283 сетапов) → **total R НИЖЕ** (+34.5 vs +68.5). Симптом: радуешься высокому avg-R и
WR, забывая про частоту. Правило: для ОТДЕЛЬНОЙ цепочки решает суммарный R (avg × частота), а
не R/сделку. Высоко-avg-R / низко-частотный вариант — кандидат в **грейд-бустер** (больший сайз
на подмножестве), НЕ в замену базового детектора. Зеркально: книжная одиночка с высокой precision
тоже умирает на expectancy (precision≠expectancy). См. [[rdrb-htf-1-1-1-high-conviction-subset]].

### Фича «последнее событие» (bulk_side) на дневном баре = lookahead, если событие внутри дня

Что было (etap_211, 2026-06-11): `bulk_side` (сторона последнего Bulkowski-сигнала) для
прогноза дня d включала сигнал, сработавший на 12h-баре ВНУТРИ дня d (00:00/12:00). Прогноз
делается as-of конца d-1 → будущее. Симптом: контекст-only «прогноз направления вперёд»
AUC **0.628** > стены 0.50; один Bulkowski-признак в топе; та же AUC на close-to-close
(разные цели, одинаковый AUC = артефакт). Фикс: `shift(1)` (только сигналы ≤ d-1) → AUC
**0.628→0.512**. Правило: фичу-«последнее событие» на дневном баре, если событие может
случиться ВНУТРИ прогнозируемого дня, сдвигать `shift(1)`. Shuffle полной модели leak НЕ
ловит — проверять контекст-only часть отдельно + sanity на альт-цели. Подозрительно высокий
AUC прогноза-вперёд (>0.55 на направлении) → первым делом искать такой leak.
См. [[lookahead-в-bulk-side-сегодняшний-сигнал-для-прогноза-дня]].

### Дневная HTF-фича (USDT.D 1d return) на 12h-баре = lookahead → раздутый AUC

Что было: etap_189 предсказывал «+5%>снятие low/high», дал OOS AUC 0.758 (выглядело
прорывом). `usdtd_1d_ret` = дневной возврат USDT.D ЗА ДАТУ бара был importance #2.
Симптом: подозрительно высокий AUC; одна HTF-фича в топе importance.
Причина: дневная свеча закрывается в 24:00, а 12h-бар (00:00/12:00) решается раньше →
дневной возврат за ту же дату содержит будущее. После фикса (брать возврат за ПРОШЛЫЙ
день, как у Hull-1d shift +1d) AUC 0.758→0.664. ~0.07 AUC было утечкой.
Правило избегания: любую HTF/дневную фичу на LTF-баре сдвигать на close HTF-свечи
(index += tf_duration) и брать asof(last closed). Подозрительно высокий AUC (>0.7 на
финданных) — ПЕРВЫМ делом искать lookahead в HTF/agg-фичах. См. также
[[htf-lookup-must-use-last-closed-bar-not-forming]]. + net_R с микро-риском (risk<0.3%)
взрывает ΣR — отсекать вырожденные стопы. Источник: [[нейро-фильтр-не-создаёт-edge-нужен-separable-signal]]

### Конструкция бэктеста раздувает net_R: вход/резолв-рассинхрон + close-based стоп + микро-зоны

Что было (zone_reactor 2026-06-10): held/broke бэктест давал +0.78…+1.4 R/tr — оказалось
артефактом. Три источника: (1) стоп проверялся по CLOSE за дальний край, а реальный стоп
срабатывает по фитилю (intrabar) → узкие зоны «удерживались» ложно; (2) касание искалось
на ТФ зоны (1d-бар), а резолв стартовал с ПОЗЖЕ-границы (12h) → пропуск окна касание→граница
с ранними стопами/целями (= мягкий lookahead); (3) микро-зоны (ширина <0.5%) → rr 20-50,
выбросы. Симптом: R/tr слишком хорош (>+0.5), но baseline-вслепую при этом норм/минус.
Причина: вход и резолв на РАЗНЫХ точках/гранулярности; стоп по close, не по фитилю.
Правило избегания: вход и стоп резолвить НА ОДНОЙ гранулярности от ФАКТИЧЕСКОГО момента
касания; стоп = intrabar low/high (SL-first консервативно), не close; флор риска/ширины
(≥0.5%). Помни: 12h.low = min(1h.low) → 1h НЕ даёт больше стопов, чем 12h при одном входе;
если held% сильно разнятся между 1h и 12h — это РАЗНАЯ логика скриптов, а не «1h честнее».
Чистая проверка: вход=close сигнальной свечи, единый резолв. Источник:
[[2026-06-10-neuro-metalabel-no-separable-edge]] (zone_touch/backtest_user_model).

### Широкая «зона» геймит held-лейбл (реакция +5% << ширины тривиальна)

Что было (zone_reactor): топ-P «сильные зоны» модели были 10-20% шириной. Лейбл held =
«+5% раньше пробоя дальнего края»; при крае на 15% дальше +5% выполняется почти всегда.
Симптом: zone_width — топ-фича; самые уверенные предсказания — на абсурдно широких зонах.
Причина: ширина зоны механически коррелирует с лейблом, когда стоп = дальний край.
Правило избегания: кап ширины зоны на реалистичный диапазон (≤3-5%) ПЕРЕД обучением;
проверять, не сидит ли «edge» в широком-зонном хвосте (killer-тест: ограничить ширину →
выживает ли AUC/net_R). Источник: [[2026-06-10-neuro-metalabel-no-separable-edge]] (zone_width_sanity).

### Нейро/ML мета-фильтр не создаёт edge — нужен separable signal

Что было: пытались обучить классификатор (HGB-GBM + MLP) отделять good/bad сетапы
на момент формирования и отсеивать плохие. Прогнали на Bulkowski-пуле (etap_178-181,
до ~250 фич) и на i-RDRB+FVG с доказанным edge (etap_182).
Симптом: resub AUC → 1.0, но purged-CV ≈ 0.47-0.59 и OOS AUC ≈ 0.5. Отсев по P
поднимает WR на доли %, но режет ΣR (отсекает прибыльные сетапы).
Причина: исход SMC-сетапа не имеет отделимой пред-входной сигнатуры сверх геометрии
(R/ATR). Мета-лейблинг усиливает primary-модель только если у winners/losers разные
распределения фич — здесь они почти одинаковы.
Правило избегания: НЕ строить learnable мета-фильтр, пока не доказано purged-CV > ~0.55
И OOS AUC > ~0.55 одновременно. resub-AUC и KFold-без-purge обманывают (perge-утечка
через перекрытие горизонтов лейбла, AFML гл.7). Если importance концентрируется в
одной геометрической фиче — это single-feature threshold (R/ATR), не задача для сети.
Всегда смотреть ΣR + per-year, не только WR/R-tr (+WR ценой −ΣR = ложный улучшатель).
Источник: [[нейро-фильтр-не-создаёт-edge-нужен-separable-signal]]

### Instant-fill simulator завышает PnL в 3-7×

Что было: etap_42 PDF использовал `simulate_fixed_rr` без ожидания касания
entry — trade фейерится мгновенно по signal_time + tf_min независимо от
текущей цены. Числа +168R BTC (etap_42 reference) → реально +42R (×4).
Симптом: PDF числа выглядят слишком хорошо vs другие симуляторы.
Reproduce: instant baseline 165R / limit baseline 41R → ×4 inflation.
Inflation factor: BTC ×4.0 / ETH ×3.3 / SOL ×4.6.
Правило избегания: backtest simulator ОБЯЗАН ждать касания entry
(limit-fill) или использовать market-at-close с пересчётом R. instant-fill
OK только для screening relative comparison, НИКОГДА для абсолютных
live-expectations. Любой WR > 60% — first candidate проверки exec model.
Источник: [[etap-42-instant-fill-3-7x-inflation]]

### Multi-shot detector добавляет 1.7-2.3× duplicate inflation

Что было: multi-shot framework (etap_98 для 1.1.1, etap_109 для 1.1.2)
собирает все (OB-htf, entry-FVG) пары в зоне. Без дедупа `(signal_time,
direction, entry)` число trades завышено ×1.7-2.3 от реальной торговли.
Симптом: 1.1.2 BTC 6.34y multi-shot baseline +726R на 2157 closed →
после дедупа 968 unique → +315R (×2.3 inflation).
Reproduce: один top-OB содержит 5-15 macro-OB кандидатов, многие
производят тот же (signal_time, entry) → дубли (max 14× на один entry).
Правило избегания: при сравнении multi-shot PnL с canonical numbers
из CLAUDE.md — учитывать factor. Multi-shot OK для relative
baseline-vs-floating comparison (oба используют ту же выборку). Для
абсолютных live-expectations — применить дедуп.
Источник: [[multi-shot-detector-2.3x-inflation]]

### Lookahead в backtest от open() текущей свечи

Что было: `signal_time = open(i+2)` — scan 1m начинался от open бара i+2.
Симптом: grid search показал WR 63.6%, +184R за 3y BTC на VIC_EVOT — нереалистично.
Причина: внутри 15m бара i+2 есть 1m свечи с `low = low(i+2)` → entry-fill
происходит мгновенно по данным, недоступным в реал-тайме.
Правило избегания: scan всегда стартует от **close** последнего известного
бара: `signal_time + tf_duration`. Любой grid search с WR > 60% на сотнях
сделок крипто-стратегии — первый кандидат на проверку lookahead.
Источник: [[lookahead-bug-в-vic-evot-backtest]]

### Удаление state/sent_signals.json вручную → лавина сигналов

Что было: пользователь удалил `state/sent_signals.json` для очистки.
Симптом: при следующем re-scan (close 1h) бот разослал 87 «свежих»
подтверждений за последние 7 дней — все подписчики получили лавину.
Причина: дедуп ключи стёрлись, prefill_silent на момент рестарта не был
включён или не покрывал full re-scan window.
Правило избегания: НЕ удалять `state/sent_signals.json` на работающем боте.
При необходимости очистить — сначала остановить бот, удалить файл,
рестартовать (тогда prefill_silent при старте отметит сегодняшнее как sent).
Для очистки старых записей — использовать ротацию по дате, не truncate.
Источник: [[prefill silent при старте]]

### HTF lookup в backtest читает FORMING bar — lookahead

Что было: в etap_36 фильтр на Hull MA(78) на 1d поверх C2 (LTF strategy
на FVG-2h trigger). `hull_trend_label(close_1d, hull_1d, ts)` использовал
`close_1d.iloc[idx]` где idx — бар содержащий ts. Этот бар на момент ts
ЕЩЁ ФОРМИРУЕТСЯ.
Симптом: BTC C2v2 RR=1.5 показывал WR 49.0% / **+101R** / 0 минусовых
лет — выглядело как breakthrough; OOS на ETH провалился (-30R / 4/4 bad
years), что раскрыло аномалию. После audit fix (использовать last closed
1d bar `idx-1`): WR 46.6% / +66R / 1 bad year на BTC. Inflation +35R / 53%.
Причина: в pandas df.index содержит OPEN times. `searchsorted(ts, "right") - 1`

известен только в момент next bar's open — это будущая информация
относительно ts. Pine `request.security(_, "1d", close)` non-repaint
возвращает close last CLOSED 1d bar, что соответствует `close[idx - 1]`
в backtest.

**стартовать с `last_closed = idx - 1`**. Все Pine `[N]` shifts применять
от last_closed (`hull[last_closed - shift]`). Любой `htf_series.asof(ts)`
без поправки на закрытость бара — RED FLAG. Helper `htf_safe_value(series, ts)`
для централизованного использования.

### Multi-bar pattern: detect at trigger, но entry должен быть на confirm_idx

### Каскад продолжается на мёртвой родительской зоне — invalidation проверяется только на L2

Что было: в `detect_with_funnel` (etap_69) и `detect_4stage` (etap_66)
стратегии 1.1.4 проверка `l2_close > L1_active_end` стояла только на L2
(OB-4h). L3 (OB-1h/2h) и L4 (FVG-15m/20m) могли формироваться **уже после
инвалидации макрозоны L1** (FVG-d/12h).
Симптом: 24 из 186 raw-сетапов (~13%) портфеля B+F+J+K имели L3.close
после смерти L1. Эти сетапы показывали WR 21.1%, total -7R, avg -0.37R/trade —
системно проигрывали, но скрывались в общей статистике. До фикса портфель
показывал WR 59.9% / +133R, после фикса WR 64.3% / +107R / +0.93R/trade.
Причина: `l3_search_end = l2_close + l3_life` без clamp по `L1_active_end`.
Если l3_life > (L1_dead - L2_close), окно поиска L3 простирается за пределы
жизни макрозоны. При `allow_multi=5` это особенно опасно — квота
«5 сетапов на одну L1» добивается мёртвыми кандидатами.
Правило избегания: при многоуровневом каскаде с TTL/инвалидацией родительской
зоны проверка валидности должна быть на **КАЖДОМ уровне**, не только на L2.
Шаблон:
```python
if l2_close > L1_active_end: continue   # level 2
if l3_close > L1_active_end: continue   # level 3 — НЕ забыть
if l4_c2_close > L1_active_end: continue   # level 4
```
Дополнительно: clamp поискового окна сверху — `min(search_end, L1_active_end)`.

### round(x, N) ≠ толерантность — для tolerance нужен bucketing

Что было: для схлопывания «почти-дубликатов» с дребезгом 0.025-0.5%
попробовали ключ дедупа `(..., round(sl/entry, 4))`. Логика «4 знака =
0.01% bin → diff < 0.01% попадёт в один ключ».
Симптом: 0 эффекта, deduped n не изменилось (Strategy 1.1.1, 158 строк
до и после).
Причина: `round(x, N)` определяет **ширину bin**, а не **толерантность**.
Два значения с diff 0.025% МОГУТ оказаться в разных bins если они на
границе: `1.014959 → 1.0150` и `1.015214 → 1.0152` (diff 0.025% → разные ключи).
Правило избегания: для семантического схлопывания близких значений
**нельзя использовать round() как threshold**. Правильный паттерн —
bucketing: сортировка по значению + последовательное объединение пока
`abs(value_i - value_first) < THRESHOLD`. См.
[[strategy-1-1-1-dedup-bucketing-tolerance]] для готового шаблона.
Также: при объединении группы — assert на критичных полях (тут `outcome`);
если они расходятся → split, легитимные разные данные.
Источник: [[strategy-1-1-1-dedup-bucketing-tolerance]]

### Нативный 3d Binance не выровнен по UTC-эпохе

Что было: `3d` лежал в `TIMEFRAMES_NATIVE` → `get_df("3d")` тянул свечи

### Узкий стоп (ORB/breakout): издержки в R велики — всегда cost-adjust

Что было: vol-gate ORB (etap_261) дал gross +0.057…+0.150 R/сделку. Соблазн назвать edge.
Стоп у ORB = диапазон IB (узкий), поэтому фикс-% издержки в R-единицах БОЛЬШИЕ:
cost_R = round-trip% / risk%. При 0.10% round-trip и стопе 1%: 0.10R/сделку — съедает
тонкий gross. Здесь стопы оказались 1.9–3.7% → cost 0.03–0.05R, edge выжил (NET BTC
+0.098), но проверка ОБЯЗАТЕЛЬНА.
Правило: любую стратегию с узким стопом (ORB, scalp, tight-SL) судить ТОЛЬКО net в R:
вычесть cost_R = (taker×2 + slippage)/risk%. На stop-входе закладывать slippage (вход по
рынку на пробое, не лимит). Тонкий gross-edge при узком стопе почти всегда мнимый.
Источник: etap_261, 2026-06-16.

Причина: precision меряет, что СОБЫТИЕ происходит (фрактал = фитиль локальный экстремум ±2 бара,
СТРУКТУРА), а не что после него цена идёт ПРИБЫЛЬНО в нужную сторону (направление/магнитуда =
монетка — стена проекта). Tight SL на экстремуме + малый пост-событийный ход = низкий/нулевой R.
Правило: НИКОГДА не выводить торговый сигнал из «predictor precision N%» без бэктеста реального
entry/SL/TP в R, cross-asset (BTC+ETH+SOL) и по годам. Occurrence-precision и expectancy — разные
вещи; высокая первая часто = детекция структуры, а не edge. Источник: backtest_v1_signal_rr, 2026-06-18.
