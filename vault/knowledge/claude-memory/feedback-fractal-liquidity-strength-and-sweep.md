# Фрактальная ликвидность — сила и sweep

Уточнения 2026-05-24 от пользователя по работе с FH/FL.

## Сила фрактала зависит от ТФ × возраст × cluster

**Старший ТФ → значимее фрактал.** Логика: за HTF-свингом накапливается больше стопов всех мастей (от scalper'ов до swing-трейдеров). LTF-фрактал = меньше стопов = слабее магнит.

**Возраст имеет U-образную зависимость:**
- **Слишком свежий** (только что сформировался): стопы ещё не накопились в большом количестве → магнит слаб.
- **Зрелый** (прошло достаточно времени, чтобы участники открылись + поставили стопы за уровнем): максимальный эффект.
- **Слишком старый** (прошло "долго"): игроки уже закрыли позиции, переставили стопы → ликвидность улетучилась, магнит снова слаб.

**Идеальной формулы для "зрелости" не существует** — судить контекстуально (TF, волатильность, активность инструмента).

**Cluster**: несколько FH/FL близко по цене (equal highs/lows) усиливают магнит — стопы концентрируются.

## Sweep — TF-relative концепт

**Sweep на ТФ X** определяется по свечам ТФ X, не по агрегации сверху/снизу:
- **Wick достаточно как правило** (close обратно за уровнем = sweep). Тело обычно НЕ требуется.
- Но это **не жёсткое правило** — зависит от ТФ и ситуации. На каких-то сетапах нужен и body, на каких-то даже временное закрепление за уровнем допустимо (см. ниже).

**Цена может закрепиться за областью ликвидности на какое-то время, чтобы собрать новые ордера** перед основным sweep'ом. Это не инвалидирует setup автоматически — это часть процесса манипуляции.

## ⚠️ Главная тонкость: HTF sweep ≠ LTF break

**Пример** (важный для multi-TF логики):
- На **1h** новый фрактал FH=X может быть **полностью пробит** (даже close > X на нескольких 1h-барах).
- На **12h** **та же самая** свеча (агрегированная из 12 × 1h) закрывается **ниже FH** на 12h — формирует sweep на 12h.

То есть: внутри одной HTF-свечи может произойти полный цикл "пробой + возврат" на LTF, но на HTF это всего один wick. **HTF sweep "проглатывает" множество LTF событий.**

**Trading-следствия:**
1. **Sweep всегда проверять на том ТФ, где торгуешь сетап** — не агрегировать вверх/вниз.
2. LTF "пробои" внутри HTF wick = шум относительно HTF-сетапа, не инвалидация.
3. "Новые" LTF-фракталы, попавшие в HTF wick, **не отменяют HTF sweep** — они часть его.
4. Когда строишь cascade-стратегии HTF→LTF, помнить: HTF setup не разрушается LTF "пробоями" внутри HTF wick.

## Why

Пользователь явно поправил две моих неясности 2026-05-24:
1. "чем старше ТФ тем более важный фрактал. Недавно образованный фрактал еще слаб ... Слишком старые стоп ордера становятся тоже слабыми ... Идеальной формулы не существует".
2. "Фитиля достаточно как правило. Все зависит от таймфрейма и ситуации. Цена может закрепиться даже за областью ликвидности на какое-то время, чтобы набрать новых ордеров. на 1h фрактал новый может быть полностью за областью ликвидности. При этом на 12h свеча закроется ниже FH".

## How to apply

- При обсуждении / детекции фракталов: считать значимость как **TF × возраст × cluster**, не "все фракталы равны".
- При построении sweep-детекторов: **wick по умолчанию** достаточно, body не обязательно, без жёстких порогов.
- При multi-TF анализе: sweep на ТФ X проверять ТОЛЬКО по свечам ТФ X. Не агрегировать. Не считать LTF-пробои инвалидацией HTF setup.
- В smc-lib `fractal/` (когда буду оформлять) — primitive хранит точку (`fractal.high` или `fractal.low`) + `tf` + `open_time` для возраста. Сила — runtime context, не атрибут примитива.

## Уточнение про LuxAlgo "Liquidity Swings" (2026-05-24)

Пользователь прислал индикатор `~/Desktop/Ликвидность.rtf` (LuxAlgo "Liquidity Swings [LuxAlgo]"). **Этот индикатор НЕ переопределяет FH/FL** — он только **измеряет накопленный объём ликвидности за уже существующими FH/FL** (touch count + volume пока зона uncrossed) + рисует ликвидность как интервал фитиля.

**Разделение слоёв (canon):**
- **Primitive layer** (`elements/fractal/` когда буду оформлять): Williams 5-bar, N=2 — точка `fractal.high` / `fractal.low`. Это и есть FH/FL.
- **Activity / strength layer** (опциональная надстройка): LuxAlgo-стиль — count + volume баров, касавшихся зоны до cross'а. Может пригодиться как метрика "созревания" зоны.

**N=14 в LuxAlgo — внутренний параметр индикатора**, не отменяет canon N=2 для FH/FL. Индикатор и наш fractal могут параллельно существовать с разными настройками.

**Когда может пригодиться:** для объективной оценки силы зоны (вместо ручного "TF × возраст × cluster"). Сохранён как reference; в текущий момент в smc-lib не оформляем.
