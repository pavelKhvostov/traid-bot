# Allow_multi: несколько сетапов на одну макрозону L1

Решение, принятое в [[2026-05-11-strategy-114-bfjk-portfolio-bug-audit]] при
разработке [[strategy-1-1-4-bfjk-portfolio]].

## Контекст

В первоначальной реализации каскадных стратегий (1.1.1, 1.1.4) детектор
работал по схеме: «для каждой макрозоны L1 — ищи каскад L2→L3→L4. Нашёл
первый — `break`, переходи к следующей L1». То есть **максимум 1 сетап
на одну макрозону**.

При исследовании 1.1.4 (etap_69 funnel diagnostic) обнаружилось:
- 916 валидных FVG-12h макрозон за 6 лет
- Из них только 41 (4.5%) дают сетап с правилом «1 сетап = 1 L1»
- В реальности на одной активной макрозоне может произойти 2-5 retest'ов

## Проблема

Один сетап на L1 — это **искусственное ограничение**:
- Не используются повторные тесты той же зоны (которые в ICT-методологии
  считаются качественными — «вторая попытка цена качественнее первой»).
- Резко ограничивает sample size: 30 сделок за 6 лет — мало для
  статистической уверенности.
- При этом каждый дополнительный retest **геометрически независим**:
  своя FVG-15m, свой OB-1h, свой entry/SL.

## Решение

Параметр `allow_multi` — сколько каскадов разрешено на одну макрозону L1.

```python
n_setups_for_this_l1 = 0
for l2 in l2_zones:
    ...
    for l3 in l3_zones:
        ...
        for l4 in fvgs_entry:
            ...
            setups.append({...})
            n_setups_for_this_l1 += 1
            if n_setups_for_this_l1 >= allow_multi:
                break  # cap per L1
        if n_setups_for_this_l1 >= allow_multi: break
    if n_setups_for_this_l1 >= allow_multi: break
```

## Эмпирическая калибровка

На chain B (FVG-12h → OB-4h → OB-1h → FVG-15m) @ RR=2.0 no_dom:

| allow_multi | n closed | WR | total | bad/yrs |
|---|---|---|---|---|
| 1 (старое) | 30 | 56.7% | +21R | 0/6 |
| 2 | 48 | 60.4% | +39R | 0/7 |
| 3 | 53 | 60.4% | +43R | 0/7 |
| **5** | **57** | **63.2%** | **+51R** | **0/7** |
| ∞ | 62 | 66.1% | +61R | 0/7 |

**WR растёт с allow_multi** — повторные retest'ы качественнее первого
входа. Это контр-интуитивно, но подтверждается на 3 цепочках (A, B, F).

## Принятое значение

**`allow_multi = 5`** — практический sweet spot:
- Sample size ≈ 2× от baseline (одиночная)
- WR улучшение почти равно ∞-limit (+6.5pp против +9.4pp у ∞)
- Минимизирует риск переусложнения

`allow_multi = ∞` теоретически лучше, но:
- Каждый дополнительный сетап даёт убывающую отдачу
- Корреляция между сетапами растёт (одна L1 = один narrative)
- При ручной торговле невозможно открывать неограниченно сделок
