# Почему CSV а не Postgres

## Контекст

При старте проекта стоял выбор хранилища для OHLCV-данных и state:
1. **CSV-файлы** (как в существующем `obxxx.py`).
2. **SQLite** локально.
3. **PostgreSQL + TimescaleDB** сразу.

## Решение

**CSV на MVP (Phase 0–4). Миграция на TimescaleDB — Phase 5 под нейронку.**

## Почему

- **Совпадает с `obxxx.py`.** Логика `normalize_df`, `update_df_incrementally`, `compose_from_base`
  уже написана и проверена на 3 годах истории. Переписывать под ORM — риск регрессии.
- **Не нужен сервер.** Бот запускается одной командой `python -m src.main`, без отдельного
  процесса БД, без миграций, без прав доступа.
- **Данные read-heavy.** Раз в час дозаписываем новые свечи, остальное время только читаем.
  Для такой нагрузки индексы БД избыточны.
- **Объём невелик.** 3 символа × 10 ТФ × 100k свечей × ~100 байт = 300 МБ. Влезает в RAM
  через `pd.read_csv` за 1-2 сек.
- **Простая миграция.** Когда Phase 5 — `neural/` потребует аналитических запросов по 10M+ точек
  и joins между signals и OHLCV, сделаем полноценный `data_loader.py` v2 с Postgres backend
  и адаптером над `core/storage.py`. Тогда переезд — точечный, не переписывание всего.

## Когда пересмотрим решение

- Количество активов выросло > 10 → CSV → TimescaleDB.
- Аналитические запросы начали занимать > 10 секунд в Python → TimescaleDB.
- Нужна параллельная запись из нескольких процессов → CSV → Postgres.
- Запускаем бот в контейнере без персистентного диска → Postgres (но мы работаем на VPS, не в K8s).

## Альтернативы, которые не выбраны

- **SQLite** — лучше чем CSV для queries, но тот же недостаток для bootstrap (прямая миграция
  из `obxxx.py` сложнее), и на Phase 5 всё равно менять на TimescaleDB — зачем два перехода.
- **Parquet** — компактнее CSV, но тяжелее для ручного просмотра (а мы сейчас на MVP отлаживаем глазами).
