# Журнал ошибок — поиск элементов и snapshot активных зон

19 ошибок, допущенных при разработке pipeline `event_detector_v*` + `snapshot_generator_v*` (проект `живой-рынок`, 2026-06-16 → 2026-06-17). Зафиксированы в реальном времени по фидбеку user'a. Текущие версии (v11 + snapshot_v6) учитывают все эти fixes — ниже история, чтобы при будущих ревизиях не наступать снова.

---

## #1 (2026-06-16) — Filter zones по center вместо overlap

**Где:** `snapshot_generator.py` фильтр candidate zones по distance от anchor price.

**Что:** Использовался center filter (`|zone_center − cp| / cp ≤ MAX_PCT`). Должен быть overlap filter (`zone_lo ≤ cp·(1+MAX_PCT) AND zone_hi ≥ cp·(1−MAX_PCT)`).

**Эффект:** В baseline_v5 (anchor 2026-06-14 07:30 МСК, $64,470) canonical OB SHORT 4h $76,730-$78,080 пропущена — center +20.06% выходит за ±20%, но zone overlap'ит на $633. Модель не видит валидный resistance.

**Why:** Зона может частично попадать в досягаемость цены — это валидный кандидат. Center filter теряет большие зоны с центром за edge.

**Канон:** zone overlap filter.

## #2 (2026-06-16) — ob_liq не консумируется fill_full на main zone

**Где:** `event_detector.py:events_ob_liq`

**Что:** ob_liq emit'ил только `born` + `liq_first_touch` (на узкой liq_marker). НЕ применял `apply_wick_fill_mitigation` на main entry zone (canon-OB area). ob_liq зоны жили «вечно».

**Эффект:** baseline_v5 на anchor 2026-06-14 07:30 МСК: active ob (16) ≪ active ob_liq (138). Subset не может быть × 8.6 больше.

**Fix:** events_ob_liq применяет `apply_wick_fill_mitigation` на `ol.zone`; emit `fill_partial`/`fill_full` для main; `liq_first_touch` остаётся как доп. событие.

## #6 (2026-06-16) ⚠ КРИТИЧНО — forward_labeler считает «уже-внутри» зоны как hit на t=0

**Где:** `forward_labeler.py:find_first_hit`

**Что:** `hit_mask = (l_slice ≤ zhi) & (h_slice ≥ zlo)` — если anchor price УЖЕ внутри зоны, первый 1m бар тривиально hit'ит → main_goal_time_hr = 0.0. Labels «загрязнены» degenerate is_inside кейсами.

**Канон:** main_goal = зона, в которую цена пришла извне.

**Fix:** filter candidates по `NOT (zone_lo ≤ anchor_price ≤ zone_hi)`.

## #7 (2026-06-16) — ML видит initial ZoI вместо active (шринкнутой) zone

**Где:** event_detector (не emit'ит active_zone в fill_partial) + snapshot_generator (не хранит shrunk bounds).

**Что:** После каждого fill_partial real active_zone шринкается (canon wick-fill Правило 2 Модели 1). Эти границы хранились ТОЛЬКО внутри `apply_wick_fill_mitigation` (state.mit_history), не пробрасывались в events → snapshot → ML.

**Эффект:** ML обучается на стейл geometry (изначальная зона на born). Если OB $59,131-$60,438 fill_partial обрезал до $60,200, ML видит $59,131-$60,438.

**Fix:** event_detector emit'ит `active_zone_lo`/`active_zone_hi` на каждый fill_partial; snapshot хранит current active_zone; ML features = active, не initial; initial оставлен как отдельные columns.

## #8 (2026-06-16) — НЕвыбор query-based архитектуры ограничивает CPU утилизацию

**Где:** мой первоначальный design `snapshot_generator_v1`.

**Что:** v1 — state-machine (active_zones dict накапливается во времени) → anchor N+1 зависит от state после N → невозможно параллелить. На PC1 14 cores использовалось 1-2 ядра.

**Правильная архитектура:** query-based — per anchor `born_ts ≤ anchor AND (retire_ts.isna() OR retire_ts > anchor)`. Каждый anchor independent → trivial parallelism.

**Эффект:** snapshot stage 30-60 мин на 1 ядре. На 14 ядрах = 5-8 мин (6-12× быстрее).

**Канон:** при выборе архитектуры pipeline думать о параллелизме сразу.

## #12 (2026-06-16) — RB BOTTOM direction mismatch → instant retire

**Где:** `event_detector_v*:scan_rb` (или `events_rb` в v1) — передача `rb.direction` напрямую в `apply_first_touch_mitigation`.

**Что:** `detect_rb` возвращает `"bottom"`/`"top"`. `apply_first_touch_mitigation` ждёт `"long"`/`"short"`. Не-`"long"` → SHORT logic (bar.high ≥ consume_level).

**Эффект:** BOTTOM RB → SHORT logic → проверяется bar.high ≥ mid-wick → ЛЮБАЯ свеча выше mid_wick retire'ит зону сразу. 57,066 BOTTOM RB retire'ились на первой же 4h после born (99.9% retire rate). TOP RB случайно работали правильно. RB=0 на anchor 2026-06-14.

**Fix:**
```python
mit_dir = "long" if rb.direction == "bottom" else "short"
state = apply_first_touch_mitigation(initial_zone=rb.zone, direction=mit_dir, consume_at_fraction=0.5, ...)
```

## #15 (2026-06-17) — fragmented labels, отсутствует cluster-level prediction

**Где:** `forward_labeler.py` + `train_baseline*.py`.

**Что:** Forward_labeler ставит label «первая зона достигнутая» = ОДНА конкретная zone (4h OB $64,142-$65,069). Но цена отрабатывает institutional level, представленный на 5+ TFs (12h ⊃ 4h ⊃ 1h ⊃ 30m ⊃ 15m). 5+ candidate rows представляют тот же level. Только одна получает label=1.

**Эффект:** model фрагментирует prediction между похожими зонами на разных TFs. Top-1 искусственно занижена.

**Архитектура fix (v4, hierarchical multi-stage):**

| Stage | Что | Input | Output |
|---|---|---|---|
| 1. Per-TF zone classifier | Учит «как цена отрабатывает каждый TF в одиночку» | per anchor × per zone features | P(zone is main_goal) per row |
| 2. Cluster aggregator | Группирует zones по overlap (institutional level) | Stage 1 scores + cluster features (n_zones, span, dominant TF) | P(cluster is main_goal) |
| 3. Combiner | Финальная: мнение зоны + мнение кластера | Stage 1 + Stage 2 + interaction | Final per-row main_goal probability |

**Cluster definition:** active zones overlapping в price (∩ ≠ ∅) или ±X% от cluster center. Cluster_id = unique institutional level.

**Канон user (2026-06-17):** «Цена отрабатывает: сперва учится на каждом ТФ отдельно, потом учится как кластер. Потом учится как её мнения отдельных зон и кластера сочетаются.»

**Cons:** Stacking требует proper CV (stage 2/3 не должны видеть data на которой stage 1 училось — purged k-fold внутри stage).

**Status:** v4, после #14 fix.

## #16 (2026-06-17) — scan_i_fvg max_b_lookahead=300 слишком мал для cross-month 4h

**Где:** `event_detector_v3.py:scan_i_fvg` параметр `max_b_lookahead=300`.

**Что:** 4h × 300 candles = 50 дней. Cross-quarter i-FVG имеют 50-100 дней gap.

**Пример (2026-06-17):** A LONG FVG 4h $69,098-$71,241 born 2026-04-08 07:00; B SHORT FVG 4h $69,548-$69,733 born 2026-06-02 19:00; gap 56 дней = 337 свечей; 300 < 337 → break → i-FVG не детектируется.

**Fix (deployed):** `max_b_lookahead = 600` (~100 дней at 4h). Универсально.

**Лучше (pending):** scale by TF — 4h=600, 6h=400, 12h=200, 1D=100.

## #17 (2026-06-17) — i-FVG arbitrary time-limit вместо canon-based stop

**Где:** `event_detector_v3.py:scan_i_fvg` — `max_b_lookahead` (band-aid #16).

**Что:** прерывание по фиксированному количеству свечей — противоречит canon scope (snapshot = active zones, не время). Канон даёт правильный stop: «A зона мертва (consumed) → нет i-FVG».

**Касается ТОЛЬКО i-FVG.** i-RDRB не затронут (4 свечи подряд).

**Правильный fix (v4):**
```python
for A in fvgs:
    state = apply_wick_fill_mitigation(A.zone, A.direction, subsequent_bars=candles[A.c3+1:])
    A_dead_at = state.consumed_at_bar if state.is_consumed else len(candles)
    for B in fvgs_after_A:
        if B.c1_idx > A_dead_at: break  # canon-based, не time
        if B.direction == A.direction: continue
        ifvg = detect_i_fvg(A, B, between=candles[A.c3+1:B.c1])
        if ifvg is None: continue
        emit
```

O(N) вместо O(N×M) — fast + canon-correct.

**Status:** #16 band-aid (600) задеплоен; #17 (canon refactor) отложен на v4.

## #18 (2026-06-17) — Per-row scope перепутан с context scope (±2% вместо ±20%)

**Где:** обсуждение spec `snapshot_generator_v4`.

**Что:** Я (claude) перепутал два уровня scope:
- **Per-row scope** = радиус, в котором zones попадают в parquet как отдельные строки (для ML ranker).
- **Context scope** = радиус для агрегатных counts (confluence density).

User ответил «±2% от current» на вопрос про context. Я применил ±2% также для per-row.

**Эффект:** Smoke output 3 active OB 4h в snapshot, но user помнил v3 baseline = 18. Сравнение:
- v8 active OB 4h всего: **124**
- v8 active OB 4h в **±2%**: 3
- v8 active OB 4h в **±20%**: **18** ✓ совпадает с v3 baseline

**Канон:** Per-row scope = ±20% от current_price. Context scope = ±2% для confluence aggregates.

**Why:** ±20% — диапазон достижимости за день-неделю; ±2% — узкий confluence radius. НЕ путать «что ML видит» (per-row) с «насколько плотно вокруг» (context).

## #19 (2026-06-17) — Invalid active_zone в simplified LTF path → ML видит бредовые границы

**Где:** `event_detector_v10.py:_wick_fill_simplified` + `emit_wick_fill` simplified branch (15m/30m).

**Что:** Когда фитиль на 15m/30m перепрыгивал границу зоны (LONG: low ≤ z_lo; SHORT: high ≥ z_hi), код emit'ил fill_partial с этим level в active_zone → invalid bounds (`a_lo > a_hi`):

```
ob  15m short: init=(7178.19, 7185.44), active=(7194.04, 7185.44)  a_lo > a_hi
fvg 15m long:  init=(7223.80, 7226.79), active=(7223.80, 7220.10)  a_hi < a_lo
```

FULL path (4h+) защищён через `shrink_mask[:end]` — фильтр shrinks BEFORE consumed. В simplified я этого не сделал.

**Эффект:**
- **172,918 fill_partial events (42.9%)** в events_v10 invalid.
- В snapshot_v5: 22% zones имели corrupted last_active.
- ML обучалась бы на бредовых границах.

Новая инкарнация #7 (#7 = emit отсутствует, #19 = emit invalid). Категория одна — semantic несогласованность active_zone.

**Fix v11:** `_wick_fill_simplified` фильтрует fill_hits СТРОГО < retire_idx. Если фитиль перепрыгнул — только retire emit'ится, без bogus fill_partial.

**Sanity v6:** assertion `last_active_lo ≤ last_active_hi` на load (raises если нарушено).

**Verify (2026-06-17):**
- v11: 0 invalid bounds (vs 172,918 в v10)
- snapshot_v6: 0 invalid (assertion passes)
- 18 OB на 4h anchor 2026-06-14 ✓ (matches v3 baseline)
